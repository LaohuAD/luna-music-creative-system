## 1. 总原则

老胡音乐不是一次性歌词生成器，而是一个音乐作品生产系统。

每首歌必须从“想表达什么”走到“怎么被唱出来”，而不是从“写几句漂亮话”开始。

### 1.1 歌词教程和示范检查

任何写给老胡、写给学员、写给读者的歌词教程和案例分析，都要先检查示范本身是否站得住。

你不能拿自己都站不住的句子去教别人写词。教程里的示范句必须先像歌词，再谈方法。

示范句最低要求：

- 有句式结构，不是散文解释。
- 有韵脚或明确的声音回环。
- 有气口，能被人唱出来。
- 有具体画面、动作、物件或声音锚点。
- 有人物态度和情绪代价。
- 有压缩能力，用较少字数承载较多信息。
- 不像 AI 高频平均素材。

当原创示范达不到以上要求时，宁愿改成“坏例诊断 + 修改方向”，或引用经典短句分析，也不要硬造平庸句子冒充作词人水平。

这部分是内部执行检查，不要原样写成对外文章里的“铁律、最高优先级、几道门”。对外写法要靠案例、对比和分析让读者自己看见判断力。

核心判断顺序：

```text
替谁说话
→ 在什么场景说
→ 平时为什么说不出口
→ 哪一句最该被记住
→ 什么曲式和风格能承载
→ 歌词如何填进这个空间
→ 人声如何把关键句唱出来
→ 生成或发布前如何打分和盲预测
→ 生成后哪些经验要回流
→ 哪些规则需要进化
```

质量评估与预测系统吸收 `cheat-on-content` 的核心闭环：打分、盲预测、生成/发布、T+N 复盘、rubric 进化。它不只服务观点视频，也不只服务歌词；凡是歌词、长文、口播、发布包装、课程化内容中包含观点、情绪立场、传播句和可验证判断，都可以纳入这套闭环。

## 2. 阶段确认制

默认每一步都要停下来让老胡确认。

写入阶段结果文件的可复制内容必须方便直接复制。候选句、歌词、style prompt、controlled lyrics、修改建议卡、交接字段和可投喂内容，默认用 Markdown 代码块包起来。文件链接、简短说明和下一步提醒可以放在代码块外。对话回复本身保持正常阅读，不要把日常沟通内容都包进代码块。

阶段产物必须写清：

- 本阶段输入是什么。
- 本阶段做了什么判断。
- 本阶段结果是什么。
- 哪些地方需要老胡确认。
- 下一步如果确认，将进入什么阶段。

只有老胡明确说“一步到位”“直接做完”“不要中途确认”时，才可以连续完成多个阶段。

## 2.1 多角色内部生成制

老胡音乐的阶段文件不应理解为“一次生成出来的一份文档”。复杂阶段默认先经过内部多角色协作，再合成给老胡。

默认顺序：

```text
题眼导演
→ 故事编剧
→ 副歌钩子师
→ 歌词修复师
→ 结构导演
→ 批判审稿人
→ 合成编辑
→ 规则沉淀员
```

每个角色只负责一种判断。上一个角色的输出是下一个角色的输入，批判和修复结果必须回填到最终阶段文件。除非老胡要求保留过程稿，否则不要为每个角色新建作品产物文件。

执行细则见：

- `02_共享资产库/05_工具流程/音乐创作多角色协作流程.md`

## 2.2 对话式阶段协议

老胡音乐不依赖前端、后端、点选表单或数据库保存。所有音乐创作都通过对话输入、阶段文件、确认记录和共享资产沉淀推进。

当前项目使用以下对话式阶段协议：

| 对话需求 | 老胡音乐阶段 | 当前执行方式 |
|---|---|---|
| 整理灵感、找选题角度、压出题眼 | 选题题眼 | `laohu_music_topic` 输出选题方向、题眼和副歌核心句 |
| 设计曲式、曲风和纯音乐提示词 | 曲式与风格 | `laohu_music_structure` 输出纯音乐 style prompt 和曲式骨架 |
| 根据已确认骨架写完整歌词 | 按曲式填词 | `laohu_lyrics_writer` 根据已确认骨架填词 |
| 直接写歌词 | 填词 | 在不走先曲后词时使用同一套填词质量门 |
| 诊断歌词、给修改建议 | 歌词优化 | `laohu_lyrics_repair` 输出问题卡和可应用改句 |
| 生成最终提示词和演唱控制歌词 | 演唱控制 | `laohu_sing_control` 输出最终 style prompt 与 controlled lyrics |
| 拆解生成结果、失败样本和发布反馈 | 审查复盘 | `laohu_music_review` 或案例经验库沉淀宏观工艺 |

### 执行替代规则

- 点选项由对话输入、老胡口述、参考材料和阶段交接字段替代。
- 输出协议由阶段文件模板和交接合约替代。
- 保存由作品目录、阶段确认记录和输入输出索引替代。
- 经验召回由共享资产库、案例经验库、归档复盘和老胡反馈记录替代。
- 阶段路由由 `laohu_music` 负责。

### 核心逻辑

必须保留这些底层判断：

- 选题题眼不是陪聊，而是把灵感压成可创作题眼。
- 题眼阶段必须判断“替谁说话”和“副歌为什么必须出现”。
- 先曲后词不是脱离内容先做曲，而是用曲式骨架框住已确认的题眼和副歌核心句。
- 歌词生成后可以轻精修，但不能暗中整首换骨架。
- 歌词优化先判断可救状态，再给问题卡。
- 演唱控制不是堆标签，而是把歌词传播资产声音化。
- 案例拆解只沉淀宏观工艺，不保存大段歌词原文。

## 3. 选题题眼阶段检查

必须回答：

- 用户表层想写什么。
- 用户真正想表达什么。
- 这首歌替谁说话。
- 明确人群画像是谁：一个具体处境里的人，而不是所有人。
- 听众从哪个场景进入。
- 听众在故事开始、中段翻转、结尾认同时分别是什么心理。
- 核心关系是谁和谁之间发生了什么变化。
- 核心关系代价是什么。
- 故事引擎是什么：事件、冲突、反差、转折如何把题眼推出来。
- 情绪债务是什么：亏欠、误解、告别、后悔、证明、释怀或重新命名里，哪一项是主债务。
- 证据配额是什么：后续歌词必须调用哪些动作、物件、时间、场景和关系代价。
- 核心画面资产是什么：哪些动作、物件、场景、声音会跨乐段反复出现或变形，而不是一次性装饰。
- 单句画面能否像视频提示词一样说清主体、场景、动作、物件、观看距离、时间变化、情绪泄露和声音锚点。
- 情绪是委屈、亏欠、想念、不甘、后悔、体面、释怀，还是混合情绪。
- 副歌要替听众压出哪一句话。
- 哪一句最可能成为传播资产。
- 整首歌从哪个事件、动作或画面引出题眼。
- Verse、Pre-Chorus、Chorus、Bridge、Outro 的歌词走向预案是什么。
- 第一次副歌、第二次副歌、最终副歌分别改变了什么听感和意义。

输出至少 3 个差异明显的方向。

每个方向必须包含：

```text
方向标题
作品简介
嘴替对象
具体场景
明确人群
受众心理
核心关系
故事引擎
情绪债务
证据配额
核心画面资产
画面感三层结构
核心情绪
题眼判断
副歌核心句候选
整首歌总纲
情绪曲线
副歌递进规划
不建议走的方向
```

禁止：

- 写成营销卡片。
- 写成抽象分析报告。
- 提前写完整歌词。
- 把“传播”理解成喊口号。
- 把所有方向都写成都市苦情或民谣叙事。
- 只写副歌候选，不规划整首歌的情绪和段落走向。
- 受众写得太宽，导致歌词谁都能听但谁都不被点名。
- 没有冲突、反差和转折，只是平铺赞美、平铺怀念或平铺人生道理。
- 没有情绪债务，导致副歌像正确总结，而不是不得不唱出来的一句话。

## 4. 题眼与副歌核心句规则

题眼不是标题，也不是口号。题眼是整首歌的材料筛选器。

副歌核心句必须做到：

- 可唱。
- 可记。
- 可复述。
- 有画面：能被描述成一个三到五秒镜头，而不是只有概念词。
- 有主体、场景、动作、物件、时间变化和声音锚点中的至少三项。
- 有人物位置。
- 有情绪代价。
- 有主韵辙、押韵模式、句尾词池和清楚的声音回环。
- 有短视频、BGM、评论区引用或现场合唱潜力。

副歌核心句可以在后续填词中因韵脚、气口、曲式稍微变化，但不能改变原本意思。

如果一句话只漂亮但不能唱，不能作为副歌核心句。

副歌韵脚必须前置：

```text
先定主韵辙
→ 再定押韵模式
→ 再定句尾词池
→ 再写副歌
→ 再做韵脚审计表
```

流行副歌最低要求：

- 核心四句至少三句落在同一主韵辙。
- 第 1 句和第 4 句必须押同一主韵或稳定近韵。
- 出辙词不得承担高潮长音、现场合唱尾音或短视频切片尾音。
- Final Chorus 可以变体，但不得突然换掉主韵系统。
- 每次输出歌词都必须附带副歌韵脚审计表。

画面感借用老胡 AI 视觉项目的提示词方法：不要写“有画面感”，要写清“谁在什么地方做了什么，哪个物件或声音让情绪落住”。副歌核心句必须能回答：

```text
第0秒：听众先看见什么？
第1到2秒：人物做了什么动作？
第3-5秒：哪个物件、声音或表情让情绪落住？
```

题眼阶段要区分核心画面资产和一次性装饰画面。核心画面资产会在 Verse、Chorus、Bridge 或 Outro 里反复出现或变形，并承载题眼和情绪债务；一次性装饰画面删掉以后不影响歌曲核心。

选题题眼还必须能指导曲式与填词。题眼不是孤立金句，而是整首歌的上游规划：它要告诉后续阶段从什么场景进、每段歌词如何推进、每次副歌出现时听众应该获得什么新感受。

执行细则见：

- `02_共享资产库/02_音乐语言资产/题眼与副歌hook库/题眼与副歌核心句规则.md`

## 5. 曲式与风格阶段检查

曲式与风格不是独立审美选择，而是替题眼找声音容器。

必须回答：

- 这首歌最适合的主风格是什么。
- 辅助风格只能起什么局部作用。
- 为什么不是其他风格。
- 这个风格如何放大题眼。
- 哪个乐段负责叙事，哪个乐段负责收紧，哪个乐段负责释放。
- 副歌核心句在哪个位置出现。
- 哪里需要 hook space。
- 哪里要给人声预留气口。

输出必须包含：

```text
主风格
辅助风格
BPM 范围
曲式结构
段落功能
能量曲线
纯音乐 style prompt
编曲取舍
后续填词空间
```

注意：

- 纯音乐阶段不生成具体人声画像。
- 可以写 vocal pocket、hook space、leave room for lead vocal。
- 风格提示词可以保留英文，因为 AI 音乐平台通常需要英文 prompt；但说明和判断用中文。
- 禁止默认 generic pop ballad、generic piano ballad、cinematic emotional song。

执行细则见：

- `02_共享资产库/02_音乐语言资产/曲式结构库/曲式结构与乐段功能规则.md`

## 6. 风格流派映射原则

风格选择必须从题材、人物、情绪、传播场景一起判断。

常见映射：

- 流行：副歌核心句短、准、可复唱，适合情绪出口和大众传播。
- 民谣：适合生活叙事、近人声、弱押、真实处境，但不能把所有现实题材都写成民谣。
- Rap / Hip-Hop：适合态度、反击、社会观察、观点刀口、bar 和韵脚密度。
- R&B：适合贴耳、身体律动、暧昧停顿、重复变体和私密情绪。
- 摇滚：适合宣泄、反抗、强动词、可喊副歌和舞台爆发。
- 电子：适合冷感、循环、切片、氛围、身体节奏和声音单位。
- 国风 / 古风：适合历史、命运、凝练意象和语体统一，禁止符号堆砌。
- 儿童 / 合家欢：适合短句、动作、重复、清楚和儿童可理解修辞。

如果用户要求多风格，必须裁决主导和辅助，不能平均硬塞。

## 7. 填词阶段检查

歌词必须服务已确认的曲式和题眼。

写词前内部确认：

- 歌词是不是站在正确的人称和听众位置。
- Verse 是否负责场景和人物进入。
- Pre-Chorus 是否负责收紧和逼出副歌。
- Chorus 是否兑现题眼和核心句。
- Bridge 是否有翻面、反问或情绪转向。
- Outro 是否有余波，而不是总结作文。

歌词必须做到：

- 可唱、可懂、可记、可剪。
- 有真人痕迹。
- 有生活碎料，但细节不是装饰。
- 有情绪拧巴，不是全程正确情绪。
- 有韵脚和唱感，但韵脚不能牺牲语义。
- 风格语言匹配，不能所有歌都写成民谣。

禁止：

- 现代诗分行冒充歌词。
- 句句正确总结。
- 公益宣传腔。
- 城市宣传语。
- 用玄词和陌生搭配冒充高级。
- 用深夜、通勤、未回复消息等高频 AI 素材冒充人味。

填词必须额外执行整首歌质量门：

- 整首歌成立：Verse、Pre、Chorus、Bridge、Outro 互相支撑，不是局部好句拼接。
- 情绪发动机：副歌是情绪被逼出来，不是主题总结。
- 真人痕迹：具体动作和生活碎料必须参与关系压力，不做装饰。
- 修辞有根：比喻、拟人、借物抒情必须有共同动作、物性或情绪方向。
- 风格语言匹配：Rap 要有 bar，R&B 要有口腔律动，摇滚要能喊，电子要能循环，国风要语体统一。
- 韵脚有语义价值：押韵不能牺牲人物自然表达。

当老胡要求“重新创作”或“完全重新创作”时，填词阶段必须按新稿处理：

```text
不是沿用原歌词做局部优化。
不是只替换副歌韵脚。
不是把原来的段落重新整理成代码块。
必须重新确定歌名、题眼表达、副歌 hook、主歌事件入口、核心画面资产、Bridge 转折和 Outro 收束。
只允许保留老胡明确要求保留的主题、风格、对象或场景约束。
```

执行细则见：

- `02_共享资产库/02_音乐语言资产/题眼与副歌hook库/题眼与副歌核心句规则.md`
- `02_共享资产库/02_音乐语言资产/韵脚与唱感库/韵脚与唱感安全规则.md`

## 8. 歌词优化阶段检查

歌词优化默认先给建议，不直接整首覆盖。

必须先判断可救状态：

```text
局部精修足够
需要结构修复
存在方向级问题，本轮只做可应用修复
```

每张修改建议卡必须包含：

```text
问题类型
原句 / 原段
建议改法
为什么这样改
这会提升唱感、人物可信度、传播力、结构发动机还是修辞准确性
老胡需要确认什么
```

禁止负优化：

- 把嘴硬修成礼貌。
- 把真诚笨拙修成标准范文。
- 把人物声口修成通用抒情腔。
- 把有刺的句子修得更顺但更弱。
- 为了押韵牺牲自然中文。

## 9. 演唱控制阶段检查

演唱控制的目标不是堆标签，而是让关键句被听见。

必须识别：

- 歌名句。
- 副歌核心句。
- 评论区引用句。
- 短视频切片句。
- 现场合唱句。
- Outro 回收句。

最终输出必须包含：

```text
最终 style prompt
controlled lyrics
section-level 控制
key-line 控制
人声策略
编曲让位策略
长度与格式检查
```

控制原则：

- line-level 控制只用于 hook、转折句、暴露句、传播锚点句和尾句。
- 不要每行都加控制。
- 不改词，除非老胡明确要求。
- 不用 generic cinematic swell 覆盖私人处境。
- 真实人味来自迟疑、气口、近距离、微哑、克制 vibrato 时，要保留。

执行细则见：

- `02_共享资产库/02_音乐语言资产/演唱控制库/演唱控制与style_prompt规则.md`
- `02_共享资产库/02_音乐语言资产/韵脚与唱感库/韵脚与唱感安全规则.md`

## 10. 审查与沉淀门槛

最终审查必须检查：

- 题眼是否兑现。
- 副歌核心句是否仍是原来的意思。
- 曲式是否承载情绪。
- 风格是否匹配人物和题材。
- 歌词是否可唱、可记、有真人痕迹。
- 演唱控制是否服务核心句。
- style prompt 是否具体，不 generic。
- 本次老胡反馈是否已经沉淀。

- `02_音乐语言资产/`
- `04_案例经验库/`
- `05_工具流程/`
- 对应 `skills/laohu_*`
- 当前作品 `09_归档复盘.md`
