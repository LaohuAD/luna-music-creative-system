# 老胡音乐作品生产系统

本文件是当前项目的唯一顶层规则入口。每次在本项目中处理音乐灵感、选题、歌词、曲式、曲风、风格提示词、演唱控制、生成复盘或经验沉淀，都优先读取并遵守本文件。

本项目不是前端项目，也不是歌词提示词收藏夹。它是老胡用对话方式进行 AI 音乐作品创作、歌词打磨、曲风设计、演唱控制和长期经验沉淀的本地项目系统。

项目明确不要前端、不要登录、不要后端、不要点选表单、不要 UI 交互。所有流程通过对话推进，文件系统负责保存作品、阶段确认、共享规则和经验资产。

## 0. 歌词教程和示范写作提醒

任何歌词教程、歌词分析、歌词生成、歌词优化和案例讲解，都要先检查示范本身是否站得住。

如果要教别人怎么写词，示范句本身必须像歌词、能唱、能记、有态度、有画面、有声音落点。不能自己写不出合格示范，却拿低水平句子教别人。

写作提醒：

- 不允许用 AI 大数据平均句、口水句、大白话说明句冒充歌词示范。
- 不允许用“深夜、地铁、未读消息、妈妈电话、工牌、电梯门、凌晨改方案”等高频素材冒充人味，除非它来自老胡真实经历且被重新写出独特角度。
- 不允许把“解释性句子”当歌词，例如只是在说明一个观点、一个情绪、一个道理，但没有韵脚、气口、句尾、动作、画面和可唱性。
- 教程里的每个示范都代表作者水平。示范写得差，就说明教程没有资格成立。
- 如果当前无法写出足够高质量的原创示范，优先引用经典歌词短句、传统诗词短句或老胡已有作品片段进行分析；引用必须短，只分析技法，不大段复制。
- 写教程时必须把自己放在真正作词人的视角，而不是 AI 工具说明书的视角。要有判断、有取舍、有审美、有锋芒。
- 所有示范句输出前必须检查：像不像歌、能不能唱、句尾能不能落、有没有画面、有没有人物态度、有没有韵脚意识、是不是普通 AI 会写出的平均句。
- 以上是内部执行检查，不要原样写进对外文章；对外文章要用案例和分析自然证明作者懂，不要用“铁律、最高优先级、几道门”这类词自证。

## 1. 项目定位

这个项目的目标，是形成一个适合老胡长期使用的对话式音乐作品生产系统。

它要解决的不是“生成一首歌词”，而是让一首歌从灵感开始，经过题眼、副歌核心句、曲式骨架、风格提示词、填词、歌词优化、演唱控制、审查复盘，逐步长成一个可生成、可发布、可复盘、可继续迭代的作品。

每次处理材料，都要同时判断四件事：

1. 它属于哪个音乐作品；如果还没有作品，就先建立作品目录或归入 `00_输入原料/`。
2. 它处在音乐作品链路哪一环：灵感、选题、题眼、副歌、曲式、曲风、填词、修词、演唱控制、生成素材、发布还是复盘。
3. 它是否能沉淀成共享资产：题眼规则、风格映射、曲式结构、韵脚策略、演唱控制、成功案例、失败案例或工具流程。
4. 它如何让下一次写歌更好，而不是只让当前这一次结果更顺。

## 2. 顶层目录

- `00_输入原料/`：还没有明确归属作品的灵感、口述、参考歌词、参考歌曲、外部文章、用户反馈、经验材料。
- `01_作品项目/`：所有正式音乐作品的唯一主入口。每首歌一个目录。
- `02_共享资产库/`：所有跨作品复用的音乐创作资产。
- `03_发布与课程化/`：把音乐作品经验转成教程、口播、课程、社群交付内容。
- `04_诊断与系统日志/`：项目级诊断、初始化记录、系统结构调整记录。
- `skills/`：项目专属 skill 或执行规则。
- `输入输出索引.md`：项目记忆入口，记录输入、产出、沉淀资产和状态。

目录用途：

- `00_项目总览.md`：歌曲目标、目标听众、核心情绪、当前阶段、下一步动作。
- `00_阶段确认记录.md`：记录每个阶段的产物、状态、老胡确认和下一阶段。
- `00_原始输入/`：老胡灵感、口述、参考歌词、参考歌曲、参考视频、原始资料。
- `01_选题题眼/`：选题角度、嘴替对象、情绪场景、副歌核心句、传播句候选。
- `02_曲式与风格/`：先曲后词第一阶段的纯音乐 style prompt、曲式结构、BPM、编曲取舍。
- `03_歌词/`：按曲式骨架填出来的歌词初稿和后续确认稿。
- `04_歌词优化/`：歌词诊断、问题卡、修改建议、采纳记录。
- `05_演唱控制/`：最终 style prompt、带 section / vocal direction / meta tags 的演唱控制歌词。
- `06_生成素材/`：AI 平台生成的音频、生成记录、失败样本、修改意见。
- `07_发布包装/`：歌名、标题、封面文案、短视频切片、发布说明。
- `09_归档复盘.md`：整首歌创作过程、老胡反馈、成功失败经验、反哺共享资产记录。

原则：

- 只要材料服务某一首歌，就放入该作品目录，不放顶层横向工作台。
- 不确定归属哪首歌的材料，先放 `00_输入原料/`，并在 `输入输出索引.md` 标记待归属。
- 每个阶段产物都要记录是否已由老胡确认。未确认阶段不能默认推进下一阶段，除非老胡明确说“一步到位”。
- 阶段交接字段和确认门统一遵守 `02_共享资产库/05_工具流程/音乐创作阶段交接合约.md`。

## 4. 标准创作链路

默认采用分阶段确认制：

```text
用户输入灵感
→ 选题角度 / 题眼 / 副歌核心句
→ 老胡确认
→ 曲式结构 / 纯音乐风格提示词
→ 老胡确认
→ 按曲式填词
→ 老胡确认
→ 歌词优化建议
→ 老胡反馈 / 采纳
→ 演唱控制和最终 style prompt
→ 老胡确认
→ 审查 / 归档 / 经验沉淀
```

除非老胡明确说“直接一口气做完”“一步到位”“不要中途确认”，否则每一步都要：

1. 输出当前阶段结果。
2. 说明本阶段做了什么判断。
3. 说明下一步应该做什么。
4. 等待老胡确认或修改意见。
5. 把老胡反馈沉淀到作品记录和共享资产。

## 5. 核心流程定义

### 5.1 选题题眼阶段

输入：一句灵感、一段口述、一个故事、一个情绪、一个人物、一个参考方向。

主体执行由 `laohu_music_topic` 完成；可补充读取：

- `02_共享资产库/02_音乐语言资产/题眼与副歌hook库/题眼与副歌核心句规则.md`

输出：

- 用户表层想写什么。
- 用户真正想表达什么。
- 这首歌替谁说话。
- 具体场景是什么。
- 核心情绪和关系代价是什么。
- 3 个差异明显的选题角度。
- 1-3 句副歌核心句 / 词眼候选。
- 哪一句最适合成为短视频、BGM、评论区引用或现场合唱入口。

注意：

- 这一步可以出现一两句歌词雏形，但不能提前写完整歌词。
- 副歌核心句可以后续因韵脚、唱感和曲式轻微变化，但核心意思不能变。
- 选题题眼不是营销标题，而是整首歌的情绪发动机。

### 5.2 曲式与风格阶段

输入：已确认的选题角度、嘴替对象、题眼、副歌核心句、情绪方向。

主体执行由 `laohu_music_structure` 完成；可补充读取：

- `02_共享资产库/02_音乐语言资产/曲式结构库/曲式结构与乐段功能规则.md`
- `02_共享资产库/02_音乐语言资产/风格流派映射库/风格流派与情绪题材映射初版.md`

输出：

- 主风格 / 辅助风格判断。
- 为什么这个题材适合这种风格。
- 曲式结构：Intro、Verse、Pre-Chorus、Chorus、Bridge、Outro 等段落功能。
- 能量曲线：哪里压住，哪里抬起来，哪里留白，哪里爆发。
- 纯音乐 style prompt。
- BPM 范围。
- 编曲取舍和人声预留空间。

注意：

- 这里进入“先曲后词”的骨架阶段，但不是脱离内容的先曲后词。
- 曲式和风格必须服务前一步确认的题眼、副歌核心句和情绪锋芒。
- 不允许默认 generic pop ballad、generic piano ballad 或 cinematic emotional song。

### 5.3 填词阶段

输入：已确认的曲式结构、风格提示词、选题题眼、副歌核心句。

主体执行由 `laohu_lyrics_writer` 完成；可补充读取：

- `02_共享资产库/02_音乐语言资产/题眼与副歌hook库/题眼与副歌核心句规则.md`
- `02_共享资产库/02_音乐语言资产/韵脚与唱感库/韵脚与唱感安全规则.md`

输出：

- 完整歌词初稿。
- 歌名建议。
- 乐段标签清楚。
- 副歌核心句在 Chorus 中兑现。
- Verse / Pre / Chorus / Bridge / Outro 各自承担明确心理动作。

注意：

- 歌词要可唱、可懂、可记、可剪。
- 不要把选题分析原文复写成歌词。
- 不要把副歌写成口号。
- 不要把所有风格都写成民谣叙事。

### 5.4 歌词优化阶段

输入：歌词初稿和老胡反馈。

主体执行由 `laohu_lyrics_repair` 完成；可补充读取：

- `02_共享资产库/02_音乐语言资产/韵脚与唱感库/韵脚与唱感安全规则.md`
- `02_共享资产库/02_音乐语言资产/题眼与副歌hook库/题眼与副歌核心句规则.md`

输出：

- 当前歌词核心问题。
- 哪些问题属于局部词句，哪些属于结构，哪些属于方向。
- 修改建议卡。
- 可应用改句。
- 修改理由。
- 老胡采纳 / 否定 / 反馈记录。

注意：

- 这一步默认先给修改意见，不直接覆盖全文。
- 如果老胡明确要求“直接改”，才输出完整修改版。
- 修改不是追求更顺、更文雅，而是更像人唱、更有题眼、更可唱、更有传播句。

### 5.5 演唱控制阶段

输入：确认后的歌词、曲式结构、风格提示词、老胡补充要求。

主体执行由 `laohu_sing_control` 完成；可补充读取：

- `02_共享资产库/02_音乐语言资产/演唱控制库/演唱控制与style_prompt规则.md`
- `02_共享资产库/02_音乐语言资产/韵脚与唱感库/韵脚与唱感安全规则.md`

输出：

- 最终可投喂 AI 音乐平台的 style prompt。
- 带 section / vocal direction / meta tags 的演唱控制歌词。
- 关键句演唱控制。
- 人声音色、距离、动态、咬字、拖音、和声、声场说明。
- 长度与格式检查。

注意：

- 演唱控制不改词，只放大歌词传播资产和情绪转折。
- line-level 控制只用于 hook、转折句、暴露句、传播锚点句和尾句，不要每行都加。
- 如果之前是纯音乐骨架，最终 style prompt 必须升级成人声版，但不得推翻已确认的曲风和曲式。

### 5.6 审查与沉淀阶段

输入：全链路产物、老胡反馈、生成结果、失败样本。

输出：

- 最终审查报告。
- 题眼是否兑现。
- 副歌核心句是否保持原意。
- 曲式是否服务情绪。
- 歌词是否可唱、可记、有真人痕迹。
- style prompt 是否服务歌词。
- 演唱控制是否突出传播资产。
- 本次经验沉淀到哪些共享资产。

## 6. 资产边界

当前项目只保留能直接服务老胡音乐创作流程的运行资产。

长期有效的音乐创作能力只允许进入两类位置：

- `skills/laohu_*`：当前项目自己的执行 skill。
- `02_共享资产库/02_音乐语言资产/`：当前项目自己的共用型规则资产，统一使用 `.md` 文件。

如果某条外部经验仍然有价值，要重新提炼成当前项目的 skill 或共享资产，不允许把外部项目文件整包留在项目里。

## 7. 学习与进化

这个项目必须能学习、成长、输入资料、吸收反馈。

当老胡提供：

- 新歌词参考。
- 优秀歌曲拆解。
- 对某句歌词的否定或肯定。
- 对风格、曲式、演唱控制的反馈。
- 生成失败样本。
- 发布数据和评论反馈。

必须判断是否要更新：

- `02_共享资产库/02_音乐语言资产/`
- `02_共享资产库/04_案例经验库/`
- `02_共享资产库/05_工具流程/`
- 对应 `skills/laohu_*`
- 当前作品 `09_归档复盘.md`

原则：

- 不要只改当前歌词，要把长期有效的经验写进执行文件。
- 不要把老胡反馈当成临时聊天。
- 不要把参考歌当成模仿对象，要拆它的宏观创作工艺。
- 不保存大段歌词原文，不保存可还原原作的连续歌词。

## 8. 结果交付规则

不管什么时候，只要本轮产出了结果文件，回复老胡时必须直接给具体结果文件链接，不能只给目录链接。

所有写入结果文件的可复制内容，例如候选句、歌词、style prompt、controlled lyrics、交接字段、修改建议卡、复盘结论、可投喂内容和可复制清单，默认都必须用 Markdown 代码块包起来，方便老胡在文件里直接复制。

结果文件里允许放在代码块外的内容：

- 简短说明。
- 文件链接。
- 下一步提醒。
- 极短的状态句。

对话回复本身保持正常阅读，不要为了这条规则把日常沟通内容都包进代码块。回复里只需要给文件链接和简短说明；如果确实在聊天窗口直接交付大段可复制结果，再把那段结果单独放进代码块。

如果同一个结果文件里既有说明又有结果，说明放代码块外，结果放代码块内。不要把大量可复制正文散落在普通 Markdown 段落或普通列表里。

默认交付规则：

- 产出了题眼方案，就给 `01_选题题眼/文本/...` 具体文件。
- 产出了曲式与风格，就给 `02_曲式与风格/文本/...` 具体文件。
- 产出了歌词，就给 `03_歌词/文本/...` 具体文件。
- 产出了优化建议，就给 `04_歌词优化/文本/...` 具体文件。
- 产出了演唱控制，就给 `05_演唱控制/文本/...` 具体文件。
- 产出了复盘，就给 `09_归档复盘.md` 或具体复盘文件。

## 9. 根本目的判断规则

处理老胡任何音乐创作需求时，不要只停留在表面说出的风格、歌词句、曲式名或局部修改点上，必须主动判断背后的根本目的。

执行时先分三层：

```text
表层需求：老胡直接说了什么。
真实目标：老胡真正想让这首歌表达什么、解决什么、传播什么。
根本目的：这次处理如何提升最终作品质量、生成可用性、发布价值和后续迭代效率。
```

如果表层指令只能解决症状，不能解决根本目的，要主动指出，并给出更接近根本目的的处理方案。

如果可以合理推断且不会显著增加风险，直接纳入方案；如果会显著改变范围、成本或需要老胡取舍，再确认。
