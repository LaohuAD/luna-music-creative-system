# 音乐创作多角色协作流程

本文件定义老胡音乐项目内部多 Agent / 多角色协作方式。

它解决的问题不是多写几个版本，而是让一个阶段产物在交付老胡之前，先经过创作、批判、修复、合成和规则沉淀，最后仍然只呈现一份干净的阶段文件。

## 1. 总原则

```text
一个阶段可以有多个内部角色
→ 每个角色只负责一种判断
→ 上一角色输出成为下一角色输入
→ 批判和修复必须回填到最终阶段文件
→ 默认不为每个角色新建用户可见产物文件
```

除非老胡明确要求保留过程稿，否则内部角色草稿不单独落成作品目录里的新文件。最终只更新当前阶段产物文件、阶段确认记录和必要的共享规则。

## 1.1 叙事依据

多角色流程不是为了形式复杂，而是为了避免 AI 一次性生成时常见的平铺、普通、无冲突和无转折。

可复用判断：

- 叙事运输理论提示：具体人物、场景、情绪和画面更容易把听众带入故事世界。歌词不能只给抽象观点，要让听众进入一个人、一件事、一个转折。
- 叙事身份理论提示：成年人回望人生时，常通过关键场景和转折点重新解释过去。写中老年回忆师恩，不能只写“当年感动”，更要写“当年没懂，后来才懂”。
- 师生关系研究提示：老师对人的影响不只在知识，也在关系、支持、情绪和后来的人生理解。歌词里应优先抓具体互动事件，如点名、批改、提醒、托住、纠正，而不是泛泛赞美职业。

落地到写歌：

```text
明确人群
→ 具体场景
→ 当时误解或无感
→ 后来被现实翻转
→ 重新理解关系
→ 副歌唱出迟来的认领
```

参考来源：

- Green & Brock, 2000，叙事运输理论：`https://pubmed.ncbi.nlm.nih.gov/11079236/`
- McAdams & Olson, 2010，叙事身份与人生故事：`https://www.annualreviews.org/content/journals/10.1146/annurev.psych.093008.100507`
- `Recalling life-changing teachers`，关于成年人回忆有改变性影响的教师关系：`https://www.sciencedirect.com/science/article/abs/pii/S0883035517309722`

## 2. 标准角色

### 2.1 题眼导演

输入：

- 老胡原始灵感。
- 已有方向选择。
- 项目总览和阶段确认记录。

职责：

- 锁定明确人群，不写成“所有人”。
- 确定嘴替对象、具体场景、核心关系和禁用路线。
- 判断这首歌到底替听众说哪一句不好意思说出口的话。

输出：

```text
明确人群：
嘴替对象：
具体场景：
核心关系：
禁用路线：
```

### 2.2 故事编剧

输入：

- 题眼导演输出。
- 老胡补充的真实心理、时代节点、人物范围。

职责：

- 设计事件入口、冲突、反差和转折。
- 模拟听众在故事开始、中段、结尾的心理变化。
- 锁定情绪债务和证据配额。
- 防止全篇平铺赞美、平铺怀念或平铺人生道理。

输出：

```text
叙事入口：
受众心理：
故事引擎：
情绪债务：
证据配额：
情绪曲线：
```

### 2.3 副歌钩子师

输入：

- 题眼导演输出。
- 故事编剧输出。
- 韵脚与唱感规则。

职责：

- 把故事压力压成 1-2 句可唱、可记、可传播的副歌核心句。
- 提前声明主韵脚。
- 保证两句式副歌核心句押同一主韵或稳定近韵。
- 不把孤立金句当确认副歌。

输出：

```text
副歌核心句候选：
主韵脚：
句尾承载：
传播理由：
风险说明：
```

### 2.4 歌词修复师

输入：

- 副歌钩子师候选。
- 题眼导演和故事编剧的上下文。
- `laohu_lyrics_repair` 的诊断标准。

职责：

- 在选题题眼阶段也可以提前调用歌词优化能力，专门打磨副歌核心句和题眼表达。
- 检查副歌是否像歌词、是否押韵、是否顺口、是否有故事压力。
- 给出局部替换句，不推翻已确认方向。

输出：

```text
可救状态：
问题卡：
建议改句：
采纳建议：
回填版本：
```

注意：这里调用的是歌词优化能力，不代表进入正式“歌词优化阶段”，也不创建 `04_歌词优化` 阶段产物，除非老胡明确要求保存过程。

### 2.5 结构导演

输入：

- 题眼导演输出。
- 故事编剧输出。
- 回填后的副歌核心句。

职责：

- 判断 Verse / Pre-Chorus / Chorus / Bridge / Outro 的心理动作。
- 规划第一次、第二次、最终副歌的意义递进。
- 给曲式与风格阶段提供明确交接字段。

输出：

```text
乐段预案：
副歌递进：
曲式调用重点：
禁用写法：
```

### 2.6 批判审稿人

输入：

- 当前阶段合成稿。

职责：

- 专门挑错，不负责讨好。
- 检查是否受众过宽、故事老套、没有冲突、没有转折、情绪平、像作文、像宣传片。
- 检查副歌是否只是漂亮普通句，是否缺少上下文，是否不能指导后续曲式和填词。

输出：

```text
主要问题：
方向级风险：
结构级风险：
副歌风险：
必须修复项：
```

### 2.7 合成编辑

输入：

- 前面所有角色输出。
- 批判审稿人的必须修复项。

职责：

- 把多角色输出合并成一份用户可读的阶段文件。
- 删除内部推演痕迹、重复段落和不必要的过程解释。
- 保留对下游有用的交接字段、确认门和关键创作判断。

输出：

```text
当前阶段最终文件：
阶段确认记录：
需要老胡确认的问题：
下一阶段交接字段：
```

### 2.8 规则沉淀员

输入：

- 老胡反馈。
- 本轮批判和修复结论。
- 当前规则文件。

职责：

- 判断哪些经验是长期有效规则。
- 及时写回 skill、共享资产、流程文件或阶段确认记录。
- 不等最终复盘才总结经验。

输出：

```text
已更新规则：
当前作品状态变化：
后续复用提醒：
```

## 3. 选题题眼阶段默认管线

```text
老胡灵感 / 反馈
→ 题眼导演：锁定人群、场景、核心关系
→ 故事编剧：设计故事引擎、情绪债务、证据配额
→ 副歌钩子师：产出押韵 hook 候选
→ 歌词修复师：按修词标准打磨 hook
→ 结构导演：规划乐段和副歌递进
→ 批判审稿人：检查平铺、无冲突、普通、不可唱
→ 合成编辑：回填到 `01_选题题眼/文本/yyyy-mm-dd_选题题眼方案.md`
→ 规则沉淀员：必要时更新 skill / 共享资产 / 阶段确认记录
```

选题题眼阶段的最终文件必须包含：

- 明确人群。
- 受众心理。
- 核心关系。
- 故事引擎。
- 情绪债务。
- 证据配额。
- 题眼。
- 副歌核心句和主韵脚。
- 乐段预案。
- 副歌递进。
- 下游曲式和填词必须调用的素材。

## 4. 中间结果的文件规则

默认不新建这些文件：

- `题眼导演草稿.md`
- `副歌优化草稿.md`
- `批判审稿意见.md`
- `第2版/第3版/打磨版.md`

允许新建内部过程文件的情况：

- 老胡明确要求保留过程。
- 自动化批量运行需要审计日志。
- 一个阶段产物已经确认，后续要保留确认稿并另起修改稿。
- 中间结果是音频、图片、视频、外部素材等不可覆盖资产。

如需保存自动化审计日志，优先放入：

```text
04_诊断与系统日志/
```

不要污染作品阶段目录的最终交付区。

## 5. 质量门

任何阶段产物交给老胡前，至少过一轮批判审稿人。

选题题眼阶段重点检查：

- 有没有明确人群，而不是泛泛写所有人。
- 有没有具体场景和可拍画面。
- 有没有核心关系变化。
- 有没有情绪债务。
- 有没有故事冲突、反差或转折。
- 副歌是不是由前文压力压出来的落点。
- 副歌是否押韵、可唱、可记。
- 乐段预案是否能指导曲式和填词。

如果批判审稿人指出方向级问题，不能靠润色解决，必须回到题眼导演或故事编剧重做。

## 6. 对老胡的呈现方式

老胡看到的应该是一份质量更高的阶段文件，而不是一堆中间草稿。

回复老胡时只说明：

```text
本轮更新了什么文件。
核心判断改成了什么。
哪些规则已沉淀。
当前还停在哪个确认门。
需要老胡确认什么。
```

不要把内部每个角色的长篇推理全部贴给老胡。角色机制是为了提高质量，不是为了增加阅读负担。
