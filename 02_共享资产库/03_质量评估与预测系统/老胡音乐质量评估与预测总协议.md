# 老胡音乐质量评估与预测总协议

本文件把 `XBuilderLAB/cheat-on-content` 的核心方法融合进老胡音乐项目。原项目是 MIT License，本项目不整包复制外部仓库，只吸收其高密度工作机制，并改写成适合音乐、歌词、观点表达、AI 生成和发布复盘的本地协议。

## 1. 核心定位

老胡音乐不是“生成一首歌”就结束。每首作品都应当被当成一次可校准实验：

```text
打分
→ 盲预测
→ 生成 / 发布
→ T+N 复盘
→ 进化 rubric / 规则 / pattern
```

这套机制不只服务歌词，也服务歌曲里的观点、情绪立场、选题题眼、口播包装、发布标题、短视频切片和课程化内容。

长文、歌词、口播、短视频只是载体。载体里面往往包含观点、判断、情绪立场和可传播句。凡是能被生成结果、用户反馈、播放/评论/收藏/转发或老胡主观点评验证的内容，都可以纳入这套闭环。

## 2. 从 cheat-on-content 融合过来的关键思想

### 2.1 每次判断都要留底

不要只说“我感觉这版不错”。创作前或投喂前，必须把判断写下来：

```text
当前版本为什么值得继续？
我预测它会在哪些地方好？
我预测它会在哪些地方失败？
如果生成失败，最可能是哪一类失败？
如果发布后没传播，最可能是哪一类判断错了？
```

留底的目的不是证明自己聪明，而是让后续复盘有基线。

### 2.2 盲预测比事后解释重要

预测必须在看到生成结果、播放数据、评论反馈之前完成。看过结果以后再解释，不叫预测，只能叫复盘或重构判断。

老胡音乐里的盲预测对象包括：

```text
歌词生成前预测：这版歌词是否可唱、可记、可生成？
AI 投喂前预测：style prompt 和 controlled lyrics 可能导致什么声音结果？
发布前预测：标题、切片、封面文案、评论区引用句可能表现如何？
课程化前预测：这首歌的经验是否能转成教程、口播、社群材料？
```

### 2.3 rubric 是工作台，不是博物馆

rubric 不是用来保存所有历史想法的仓库，而是当前最有用的评分公式和判断维度。

规则：

```text
被数据或反馈推翻的观察，要删除或移出当前规则。
已经被吸收成正式维度的观察，不要继续作为观察重复存在。
尚未验证的观察，放入待验证假设。
历史演化写进版本 memo、阶段确认记录或复盘，不污染当前 rubric。
```

### 2.4 升级必须有刹车

不要因为一次失败就大改公式。也不要为了让这一次看起来合理，临时放宽标准。

rubric 升级分三类：

```text
边际细化：
不改公式，只把某个维度的判分标准说得更清楚。可以快速落地。

结构升级：
新增/删除维度，或改变权重、公式、门槛。必须有多个样本或极强反例支持。

桶/预测边界重校：
不改维度，只调整“什么算好、什么算爆、什么算失败”的结果分层。
```

### 2.5 候选池要标准化

灵感、热点、参考歌、评论反馈、观点句、课程题材，都可以进入候选池。但进入候选池前必须标准化：

```text
id：
标题：
来源：
快照文本：
录入时间：
类别：
状态：
粗评分：
预测理由：
备注：
```

这能避免“看过很多东西，但不知道为什么最后选了这个”。

## 3. 老胡音乐五阶段质量闭环

### 阶段 A：Score / 打分

在正式推进前，对当前版本打分。

适用对象：

```text
选题题眼
副歌核心句
完整歌词
style prompt
controlled lyrics
发布标题
短视频切片句
课程化口播稿
```

要求：

```text
只打分，不预测。
只输出当前 rubric 下的问题，不写入正式预测日志。
如果分数低，不要靠“后面再修”推进下一阶段。
```

### 阶段 B：Blind Predict / 盲预测

在生成或发布前写预测日志。

预测必须包含：

```text
输入快照
rubric 分数
核心判断
预测结果桶
概率分布或风险分布
推理因素
锚点对比
反事实场景
关键校准假设
```

老胡音乐可以不强制所有作品都走完整预测，但凡要积累长期判断力的作品，都建议走。

### 阶段 C：Generate / Publish / 生成或发布

音乐项目里的“发布”不只是平台发布，也包括：

```text
AI 音乐平台生成
导出音频
给老胡听
发朋友圈或短视频
发公众号 / 小红书 / 视频号
做课程化拆解
```

每次进入这一阶段都要登记：

```text
投喂版本：
平台：
时间：
是否改过歌词：
是否改过 style prompt：
是否改过 controlled lyrics：
生成 / 发布链接：
```

### 阶段 D：Retro / 复盘

复盘不是“好听 / 不好听”的感受记录，而是对预测的清算。

必须回答：

```text
预测里哪些被验证？
哪些被推翻？
失败属于选题、题眼、副歌、韵脚、曲式、风格、演唱控制、平台生成还是发布包装？
有没有评论区高频引用句？
有没有用户听完记住的句子？
有没有 AI 生成反复失败的段落？
哪条观察应该进入待验证假设？
哪条经验可以直接写入规则？
```

### 阶段 E：Evolve / 进化

复盘后要决定写回哪里：

```text
当前作品阶段确认记录
当前作品 09_归档复盘.md
02_共享资产库/02_音乐语言资产/
02_共享资产库/03_质量评估与预测系统/
02_共享资产库/04_案例经验库/
02_共享资产库/05_工具流程/
对应 skills/laohu_*
```

原则：

```text
只属于当前作品的，写入当前作品。
能跨作品复用的，写入共享资产或 skill。
被验证为错误的旧规则，删除或改写，不要堆在当前规则里。
```

## 4. 老胡音乐三条不可妥协原则

### 原则 1：预测必须早于结果

看过生成音频、平台数据、评论反馈以后，不允许再写“预测”。只能写：

```text
复盘
重构判断
事后解释
```

如果用户要求“已经听完了，你再预测一下为什么会失败”，要标为复盘，不得标为盲预测。

### 原则 2：规则升级不能只凭一次情绪

老胡的强反馈必须重视，但要区分：

```text
执行错误：立即修 skill 或流程。
质量门缺失：立即补共享规则。
审美偏好：先记录，等多次验证后再升为通用规则。
方向性失败：进入案例经验库，不一定马上改公式。
```

### 原则 3：当前规则要保持干净

当前规则文件不是历史档案。

```text
历史过程写阶段确认记录。
失败样本写案例经验库。
版本升级原因写 memo 或复盘。
当前执行文件只放现在要执行的规则。
```

## 5. 文件体系映射

老胡音乐不照搬 cheat-on-content 的目录，但吸收它的职责分层。

```text
cheat-on-content rubric_notes.md
→ 老胡音乐：质量 rubric 文件

cheat-on-content predictions/
→ 老胡音乐：作品内 06_生成素材/文本/ 或 07_发布包装/文本/ 的预测日志

cheat-on-content videos/report.md
→ 老胡音乐：06_生成素材/文本/生成复盘 或 07_发布包装/文本/发布复盘

cheat-on-content rubric-memo.md
→ 老胡音乐：02_共享资产库/03_质量评估与预测系统/规则进化备忘录.md

cheat-on-content candidates.md
→ 老胡音乐：00_输入原料/ 或 作品 01_选题题眼/文本/ 的候选池文件

cheat-on-content script_patterns.md
→ 老胡音乐：02_共享资产库/04_案例经验库/ 或 02_共享资产库/05_工具流程/ 的写法 pattern
```

## 6. 当前不做的事情

```text
不安装 cheat-on-content 原 skill。
不复制外部仓库目录。
不引入前端、后端或数据库。
不把观点视频 rubric 原样当成音乐评分公式。
不把播放量预测伪装成音乐质量预测。
```

当前融合目标是：保留它的方法密度和闭环结构，让老胡音乐项目也能“判得更准”。
