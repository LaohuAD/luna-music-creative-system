# 老胡音乐作品生产系统

一个面向 AI Agent 的对话式音乐创作工作流项目。

它不是歌词模板合集，也不是音乐生成平台。它的作用是把一句灵感，逐步推进成一首可生成、可修改、可复盘、可长期进化的音乐作品。

整个系统通过 Markdown 文件、作品目录、阶段确认记录和项目专属 skill 工作，不需要前端、后端、登录系统或表单页面。

## 适合谁

- 想用 AI 辅助写歌，但不想只得到一版随机歌词的人。
- 想把灵感、题眼、副歌、曲式、曲风、歌词、演唱控制分阶段打磨的人。
- 想长期沉淀个人音乐创作方法，而不是每次重新解释偏好的人。
- 想让 AI Agent 在本地项目里按固定规则协作创作的人。
- 想独立发作品、写出个人特色，而不是批量套模板的人。

## 为什么它看起来不是一个傻瓜式成品

这个项目故意保留了一定的“粗糙度”。

这里的粗糙，不是指不重视质量，也不是说系统不会继续打磨。它更像一个给创作者使用的初始工作台：规则、流程、目录和质量门已经搭好，但最终审美、题材偏好、歌词声口、画面资产和判断标准，需要使用者在真实作品里继续训练。

如果我把所有东西都按老胡自己的审美打包成一个开箱即用的傻瓜式项目，结果会有两个问题：

1. 后面的人写出来的歌词，很容易都像“老胡模板”刻出来的。
2. 很多只想批量发布、快速摘桃子的人，可以直接拿去量产，而不是形成自己的创作判断。

所以这个项目更适合：

- 想长期独立发作品的人。
- 想写高质量歌词，而不是批量生产素材的人。
- 想形成个人语言、个人审美、个人题材系统的音乐人。
- 愿意把自己的创作方法、失败样本、发布反馈不断喂回系统的人。

它不是“拿来就自动写出爆款歌”的机器，而是一个可以慢慢长成你个人创作系统的骨架。

## 这个项目能做什么

### 1. 从灵感提炼选题题眼

输入一句灵感、一段口述、一个故事或一个情绪，系统会帮助整理：

- 表层需求：你直接说了什么。
- 真实目标：你真正想让这首歌表达什么。
- 嘴替对象：这首歌替谁说话。
- 具体场景：听众从哪个画面进入。
- 核心关系：谁和谁之间发生了什么变化。
- 情绪债务：亏欠、误解、告别、后悔、释怀等哪一笔最关键。
- 副歌核心句：哪一句最适合被记住、转发、合唱或剪成短视频。

### 2. 设计曲式和音乐风格

在题眼和副歌核心句确认后，系统会进入曲式与风格阶段，输出：

- 主风格和辅助风格。
- BPM 范围。
- Intro / Verse / Pre-Chorus / Chorus / Bridge / Outro 的段落功能。
- 能量曲线。
- 纯音乐 style prompt。
- hook space 和人声预留空间。

### 3. 按曲式写完整歌词

系统不是直接把观点改写成押韵分行，而是根据已确认的题眼、核心画面资产、主风格和曲式骨架填词。

重点检查：

- 歌词是否像人唱的话。
- 副歌是否兑现题眼。
- 句尾是否可唱。
- 韵脚是否连续。
- 是否有具体画面、动作、物件和声音锚点。
- Verse / Pre / Chorus / Bridge 是否各自承担明确功能。

### 4. 诊断和优化歌词

如果已有歌词，系统可以输出歌词优化建议：

- 哪些问题是局部词句问题。
- 哪些问题是结构问题。
- 哪些问题是方向级问题。
- 副歌是否不押韵、不好唱、没有记忆点。
- 哪些句子像 AI 平均句、说明句或文案句。
- 可直接替换的修改建议。

### 5. 生成演唱控制和最终 style prompt

当歌词确认后，系统会生成可投喂 AI 音乐平台的最终内容：

- 最终人声版 style prompt。
- 带 section-level 控制的歌词。
- 少量 key-line 演唱控制。
- 人声音色、距离、动态、气口、咬字、和声、声场说明。
- 哪些句子需要停顿、拖音、重复或让伴奏让位。

### 6. 审查、复盘和规则进化

系统会把创作和生成结果回流成长期经验：

- 题眼是否兑现。
- 副歌核心句是否保留原意。
- 曲式是否服务情绪。
- 歌词是否可唱、可记、有真人痕迹。
- style prompt 是否具体。
- 演唱控制是否突出传播资产。
- 本次失败或成功是否需要沉淀到共享规则。

### 7. 质量打分、盲预测和发布评审

系统内置了一套质量评估与预测机制，用来避免“事后觉得自己早就知道”。

它可以在生成或发布前记录：

- 当前选题题眼是否成立。
- 副歌核心句是否可唱、可记、可传播。
- 歌词是否有画面、人物、韵脚和真实声口。
- style prompt 是否足够具体。
- controlled lyrics 是否会帮助 AI 生成，而不是干扰生成。
- 发布标题、切片句、评论区引用句可能表现如何。
- 这版作品最可能失败在哪里。

核心闭环是：

```text
打分
→ 盲预测
→ 生成 / 发布
→ T+N 复盘
→ 进化 rubric / 规则 / pattern
```

这套东西不只是给歌词用，也可以给发布包装、短视频切片、课程化口播和观点表达使用。

## 核心流程

默认采用分阶段确认制：

```text
灵感输入
→ 选题角度 / 题眼 / 副歌核心句
→ 人工确认
→ 曲式结构 / 纯音乐风格提示词
→ 人工确认
→ 按曲式填词
→ 人工确认
→ 歌词优化建议
→ 人工反馈 / 采纳
→ 演唱控制和最终 style prompt
→ 人工确认
→ 审查 / 归档 / 经验沉淀
```

除非你明确说“一步到位”“直接做完”“不要中途确认”，否则系统不会默认跳过确认门。

这个设计是为了避免一个常见问题：上一阶段的题眼、副歌核心句还没打磨好，AI 就急着进入下一阶段，导致后面的曲式、歌词和演唱控制全部建立在不稳的基础上。

## 如何让系统升级

这个项目最重要的能力，不是第一次生成，而是持续吸收经验。

你可以把这些材料丢进项目：

- 自己写过的歌词。
- 你喜欢的歌词创作方法。
- 优秀歌曲拆解。
- 参考歌的宏观工艺。
- 某次 AI 生成失败的音频和原因。
- 听众评论、收藏、转发、引用句。
- 发布标题和短视频切片的表现。
- 你对某句歌词的肯定、否定和原因。
- 你自己的口头创作经验。

然后让 AI Agent 按这个流程吸收：

```text
原始材料归档
→ 判断材料类型
→ 提炼它对音乐创作哪一环有用
→ 判断是当前作品经验还是跨作品规则
→ 更新作品复盘或共享资产
→ 必要时更新对应 laohu_ skill
→ 在输入输出索引记录
```

你可以这样对 Agent 说：

```text
这是一段我认可的歌词创作方法。
请不要直接照抄原文，先判断它对题眼、副歌、韵脚、画面感、曲式还是演唱控制有用。
如果是长期有效规则，请更新到对应共享资产或 skill。
```

也可以这样说：

```text
这是一次生成失败样本。
请分析失败原因：是题眼、副歌、韵脚、曲式、风格、style prompt 还是演唱控制的问题。
如果这个问题以后还会出现，请沉淀成规则。
```

### 哪些内容应该写回哪里

| 材料 | 建议写入 |
|---|---|
| 单首歌的过程反馈 | 当前作品 `00_阶段确认记录.md` 或 `09_归档复盘.md` |
| 跨作品通用的题眼方法 | `02_共享资产库/02_音乐语言资产/题眼与副歌hook库/` |
| 押韵、句尾、气口、唱感经验 | `02_共享资产库/02_音乐语言资产/韵脚与唱感库/` |
| 曲式、风格、编曲空间经验 | `02_共享资产库/02_音乐语言资产/曲式结构库/` 或 `风格流派映射库/` |
| style prompt、controlled lyrics 经验 | `02_共享资产库/02_音乐语言资产/演唱控制库/` |
| 评分、预测、发布复盘经验 | `02_共享资产库/03_质量评估与预测系统/` |
| 已验证成功或失败的作品样本 | `02_共享资产库/04_案例经验库/` |
| 执行流程中的错误和改进 | `skills/laohu_*` 或 `02_共享资产库/05_工具流程/` |

原则很简单：只属于当前作品的，留在当前作品；能让下一首歌更好的，写进共享资产或 skill。

## 质量评估与发布评审怎么用

在正式生成或发布前，建议做一次评审。

### 生成前评审

```text
请对这版歌词、style prompt 和 controlled lyrics 做生成前评审。
输出 rubric 打分、最可能的生成失败点、是否建议投喂。
不要事后解释，要先做盲预测。
```

重点看：

- 副歌是否有稳定韵脚。
- AI 音乐平台是否容易唱清楚。
- style prompt 是否过于 generic。
- controlled lyrics 是否标签过密。
- 核心传播句是否被声音让位。

### 发布前评审

```text
请做发布前评审。
评估歌名、标题、短视频切片句、评论区引用句和封面文案。
给出预测，不要只给优化建议。
```

重点看：

- 听众第一眼知道这首歌替谁说话吗？
- 标题是否承接题眼，而不是标题党。
- 短视频 8-15 秒切片是否有清晰入口。
- 评论区有没有可以自然引用的一句话。
- 这版发布最可能被误解在哪里。

### 复盘时清算预测

生成或发布后，不要只说“效果好 / 不好”。要回到之前的预测：

```text
请复盘这次生成 / 发布。
逐条清算哪些预测被验证，哪些预测被推翻。
把可复用经验写回规则，把只属于这首歌的经验写入作品复盘。
```

## 快速开始

### 1. 获取项目

这个仓库有两种用法：一种是安装 Agent skills，另一种是获取完整项目模板。

方式 A：只安装 skills。

```bash
npx skills add LaohuAD/luna-music-creative-system
```

只想安装某一个 skill：

```bash
npx skills add LaohuAD/luna-music-creative-system --skill laohu_music
```

这会把 `skills/*/SKILL.md` 安装到当前 Agent 的 skills 目录。适合你已经有自己的项目，只想复用老胡音乐创作能力。

注意：`npx skills add` 主要安装 skill 本体。完整的作品目录、共享资产库、模板库和检查脚本仍在这个仓库里。如果你想使用完整音乐作品生产系统，建议用下面的克隆或 `degit` 方式获取整个项目。

方式 B：克隆完整项目。

```bash
git clone https://github.com/LaohuAD/luna-music-creative-system.git
cd luna-music-creative-system
```

方式 C：用 `npx degit` 拉一份干净模板。

```bash
npx degit LaohuAD/luna-music-creative-system my-music-system
cd my-music-system
```

如果你想保留 `00_输入原料/`、`01_作品项目/`、`02_共享资产库/`、`skills/` 这整套目录结构，推荐用方式 B 或 C。

### 2. 检查项目结构

```bash
bash skills/laohu_music/scripts/check_laohu_music.sh
```

如果输出：

```text
老胡音乐项目检查通过
```

说明核心目录、规则文件和脚本都在。

### 3. 创建一首新作品

```bash
bash skills/laohu_music/scripts/create_music_project.sh "我的歌名"
```

也可以指定日期：

```bash
bash skills/laohu_music/scripts/create_music_project.sh "我的歌名" "2026-06-09"
```

脚本会在 `01_作品项目/进行中/` 下创建一套完整作品目录。

### 4. 和 AI Agent 对话创作

在支持项目上下文和本地文件读写的 AI Agent 里打开本项目，然后直接输入你的灵感。

示例：

```text
我想写一首流行歌，灵感是：看人之短，天下无可交之人；看人之长，世间处处皆是吾师。所有相遇皆是渡我。
```

系统应该先进入选题题眼阶段，而不是直接写完整歌词。

## 推荐使用方式

### 新作品创作

```text
我想写一首歌，灵感是：……
请按老胡音乐流程先做选题题眼，不要直接写歌词。
```

### 确认方向后继续

```text
我确认方向 B，但副歌核心句还不够好。
请先打磨题眼和副歌核心句，不要进入曲式。
```

### 进入曲式与风格

```text
题眼和副歌核心句确认。
请进入曲式与风格阶段，输出主风格、BPM、曲式结构和纯音乐 style prompt。
```

### 按曲式填词

```text
曲式与风格确认。
请根据已确认的题眼、副歌核心句和曲式骨架，写完整歌词初稿。
```

### 优化已有歌词

```text
请诊断这版歌词。
重点看副歌韵脚、可唱性、画面感和 AI 味。
先给修改建议，不要直接整首覆盖。
```

### 生成最终投喂内容

```text
歌词确认。
请输出最终 style prompt 和带演唱控制的 controlled lyrics。
```

## 单首作品目录

每首正式作品推荐结构：

```text
<作品名>/
├── 00_项目总览.md
├── 00_阶段确认记录.md
├── 00_原始输入/
├── 01_选题题眼/
├── 02_曲式与风格/
├── 03_歌词/
├── 04_歌词优化/
├── 05_演唱控制/
├── 06_生成素材/
├── 07_发布包装/
└── 09_归档复盘.md
```

核心原则：

- 一个作品的材料尽量都放回该作品目录。
- 同一阶段的普通迭代默认覆盖当前阶段文件，不堆一堆“修改版”“最终版”“再改版”。
- 已确认的阶段可以保留确认稿；未确认阶段不要默认推进。
- 音频、图片、视频等不可覆盖素材可以单独归档。

## 创作质量原则

这个项目强调几条底层判断：

- 题眼不是标题，而是整首歌的情绪发动机。
- 副歌核心句不是口号，必须可唱、可记、押韵、有画面。
- 曲式和风格必须服务题眼，不能默认套通用抒情流行。
- 歌词不能只是观点分行，必须有动作、物件、声音锚点和人物处境。
- 韵脚不是最后修饰，副歌确认前就要设计主韵和句尾。
- 演唱控制不是堆标签，而是让关键句被听见。
- 老胡反馈不是临时聊天，长期有效经验要及时写回规则。

## 维护规则
