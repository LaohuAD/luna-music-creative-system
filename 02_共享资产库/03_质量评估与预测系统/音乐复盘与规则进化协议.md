# 音乐复盘与规则进化协议

本文件借鉴 `cheat-on-content` 的 Retro、Observation Lifecycle 和 Bump Validation Protocol，改写为老胡音乐项目的复盘与规则进化机制。

## 1. 复盘不是总结，是清算

复盘必须对照预测或阶段判断，不能只写“效果不错 / 不好”。

```text
预测了什么？
实际发生了什么？
哪些判断被验证？
哪些判断被推翻？
是单次现象，还是跨作品规律？
应该写回哪里？
```

## 2. 复盘对象

```text
AI 生成音频
歌词打磨过程
style prompt 投喂结果
发布数据
评论区反馈
老胡听后评价
课程化转化效果
```

## 3. 复盘模板

```text
复盘对象：
关联作品：
关联阶段：
预测文件：
生成 / 发布时间：
复盘时间：
数据来源：

### 实际结果

### 哪些预测被验证

### 哪些预测被推翻

### 问题归因

| 问题 | 归因层级 | 证据 | 是否可复用 |
|---|---|---|---|
|  | 选题 / 题眼 / 副歌 / 韵脚 / 曲式 / 风格 / 演唱控制 / 平台 / 发布包装 |  |  |

### 新观察

### 待验证假设

### 规则写回

### 不写回的临时现象
```

## 4. 观察生命周期

```text
单次观察
→ 待验证假设
→ 跨作品观察
→ 规律沉淀
→ 正式规则 / 维度 / skill
```

### 单次观察

只由一次作品、一次生成或一次老胡反馈支持。

处理：

```text
写入当前作品阶段确认记录或 09_归档复盘。
如果可能跨作品复用，进入待验证假设。
不要立刻写成强规则，除非是明确执行错误或硬质量门缺失。
```

### 待验证假设

有可能跨作品成立，但证据不足。

处理：

```text
写入 02_共享资产库/03_质量评估与预测系统/规则进化备忘录.md。
后续作品主动验证。
```

### 跨作品观察

至少两个作品或两个生成样本出现同一模式。

处理：

```text
进入案例经验库。
可以补充到共享资产 md 的方法解释。
暂不一定改正式规则或 skill。
```

### 规律沉淀

多次验证或一次强反例足够明确。

处理：

```text
写入对应共享资产、skill 或核心规则。
从待验证假设里移除。
```

## 5. Bump 触发条件

允许提议升级 rubric 或规则：

```text
同一类失败连续出现 3 次。
某个评分维度长期高估或低估。
某个缺失维度导致严重误判。
生成结果反复证明某类 style prompt 写法失败。
发布反馈反复证明某类副歌句传播弱。
老胡明确指出某条是长期执行规则，不是单次偏好。
```

不允许：

```text
为了当前作品过关，降低门槛。
只凭一次普通失败改权重。
把单次审美偏好升成通用规则。
为了保留历史，把被推翻的观察继续放在当前规则里。
```

## 6. Bump 必须写清楚

```text
旧规则：
新规则：
变化类型：边际细化 / 结构升级 / 公式调整 / 阶段门新增 / 禁用项新增
触发证据：
适用范围：
不适用范围：
可能副作用：
需要回测的旧作品：
```

## 7. 清算原则

每次规则升级后必须清理：

```text
被新规则吸收的观察，从待验证区删除。
被推翻的观察，从当前规则删除，必要时写入失败案例。
仍未解决的观察，保留为待验证假设。
历史过程写进复盘，不堆在当前规则里。
```

## 8. Blind 信息隔离

如果以后建立盲评分子流程，必须注意：

```text
盲评分只能读当前 rubric 和待评分文本。
不能读生成结果、发布数据、评论反馈、复盘结论。
包含真实表现的数据只放进复盘或 memo，不放进盲评分读取的当前 rubric。
```

当前项目暂不强制实现 hook，但执行时要遵守这个信息隔离原则。
